# LDS Editorial Methodology

## 1. 文書の目的

この文書は、Lodestone Design System（LDS）の設計原則を
実際のLodestone日記へ適用するための執筆・推敲手順を定める。

対象は、原文の論旨を保ちながら、
文章構造、情報の役割、視線誘導、可読性を段階的に整える編集作業である。

本書は次の問いを扱う。

> 原文をどの順序で分析し、推敲するか
> 各編集段階で何を判断するか
> どの状態をもって次の段階へ進むか
> Lodestone上の実表示をどう確認するか

---

本書は、記事へLDSを適用する編集手順と判断基準を扱う。

| Document | 本書との関係 |
|---|---|

| `01_LDS_Governance.md` | 変更、証拠、Case Study、文書ライフサイクルの規則を参照する |

- 正本、命名、Version、Statusなどの管理規則

- 個別記事のCase Study本文や検証記録
- 著者の主張、価値判断、結論そのものの決定

## 3. 編集開始前の確認

編集前に、少なくとも次を確認する。

- 原文
- 記事の目的
- 想定読者
- 著者が伝えたい中心命題
- 変更してよい範囲
- 維持すべき語調や表現
- 参照資料と出典
- Lodestone上の技術的制約

著者の意図が不明な箇所は、推測で補完しない。

事実関係、主張の強さ、批判の方向性、最終結論を変える可能性がある場合は、
編集を確定する前に著者へ確認する。

---

## 4. 編集原則

### 4.1 著者の論旨を保つ

編集は、著者の主張を別の主張へ置き換える作業ではない。

原則として、次は著者が決定する。

- 事実認識
- 評価
- 立場
- 批判の方向性
- 提案
- 結論

編集者は、これらが読者へ正確に伝わるよう構造と表現を整える。

### 4.2 装飾より先に構造を整える

文字サイズ、太字、色、折り畳みを加える前に、
論理構造、段落、見出しを確定する。

構造上の問題を装飾だけで補わない。

### 4.3 視覚表現へ意味を持たせる

視覚表現は、単に目立たせるためではなく、
情報の役割と階層を読者へ伝えるために使う。

同じ役割には同じ表現を使い、
異なる役割を同じ表現へ無理に統合しない。

### 4.4 必要最小限の強調を使う

強調が多いほど読みやすくなるとは限らない。

文字サイズ、太字、色、余白を重ねる場合は、
それぞれが必要かを個別に確認する。

### 4.5 実表示を最終判断とする

Markdown上で整っていても、Lodestone上で読みやすいとは限らない。

技術的な可否は `04_LDS_Lodestone_Authoring_Reference.md` を参照し、
最終判断はLodestoneの実表示で行う。

## 5. Editorial Workflow

編集は、次の順序で段階的に行う。

```text
編集条件の確認
↓
論理構造の解析
↓
段落と構成
↓
見出し
↓
TypographyとEmphasis
↓
Color
↓
Disclosure / Folding
↓
BBCode実装
↓
実画面確認
↓
著者確認と最終推敲
```

後の段階で前提の問題が見つかった場合は、
装飾だけを調整せず、必要な段階まで戻って修正する。

### Pass 0：編集条件の確認

記事の目的、読者、中心命題、変更可能範囲を確認する。

この段階では装飾を始めない。

完了条件：

- 記事の目的を一文で説明できる
- 著者が保持したい主張と語調が分かる
- 未確認事項と確認先が分かる

### Pass 1：論理構造の解析

本文を意味単位へ分け、各部分の役割を分類する。

基本分類：

- Fact：客観的事実、仕様、制度、数字
- Claim：他者の主張、引用、反対意見
- Analysis：筆者の疑問、検証、考察
- Conclusion：筆者の結論、着地点

必要に応じて次を補助分類として使う。

- Example：具体例
- Note：補足
- Proposal：提案
- Transition：話題転換
- Summary：要約

分類名を記事本文へ表示する必要はない。
分類の目的は、論理上の役割を編集者が見失わないことである。

完了条件：

- 中心命題が特定されている
- 前提、根拠、分析、結論の関係が説明できる
- 役割が不明な段落を把握している

### Pass 2：段落と構成

意味単位に基づいて、記事全体の順序と段落を整える。

- 一つの段落へ役割を詰め込みすぎない
- 話題が変わる箇所で段落を分ける
- 前提を示してから分析へ進む
- 結論に必要な根拠を先に配置する
- 重複する説明を統合する
- 本筋から外れる補足を識別する

空行は装飾ではなく、意味の境界として扱う。

完了条件：

- 段落の順序だけで論理が追える
- 各段落の役割を短く説明できる
- 不要な重複と唐突な話題転換がない

### Pass 3：見出し

見出しは、本文の飾りではなく要約として設計する。

見出しだけを順に読んでも、
記事の主題、論点の展開、結論の位置が把握できる状態を目指す。

- 見出し階層と論理階層を一致させる
- 内容を持たない抽象的な見出しを避ける
- 同じ階層では粒度と文体を揃える
- 見出しを過度に増やして本文を分断しない

完了条件：

- 見出しだけで記事の概要を説明できる
- 階層の飛びや同格でない見出しがない

### Pass 4：TypographyとEmphasis

文字サイズと太字で、情報階層と文章の声量を整える。

- 見出しは本文より明確に強くする
- 同じ階層では同じ表現を使う
- 長文全体を拡大しない
- 太字は短い語句や命題へ限定する
- 見出しより本文の強調を強くしない
- 色を使う前に、文字サイズと太字だけで階層が伝わるか確認する

重要な帰結は、短い命題と標準的な本文説明に分けると、
流し読みと精読の両方へ対応しやすい。

具体的なタグ構文と利用可能な値は、
`04_LDS_Lodestone_Authoring_Reference.md` を参照する。

完了条件：

- 色がなくても主要な階層を判別できる
- 強調箇所が記事の中心命題と対応している

### Pass 5：Color

色は重要度だけでなく、情報の役割を示すために使う。

暫定的な意味対応は次の通り。

| 情報の役割 | 色の方向性 |
|---|---|
| 見出し、構造 | シアン系 |
| 事実、仕様、制度、影響 | 緑系 |
| 他者の主張、引用、反対意見 | 紫系 |
| 筆者の分析、考察 | 水色系または色なし |
| 筆者の結論、着地点 | 低彩度の暖色系 |

これは編集上の意味対応であり、
Lodestoneの技術仕様や固定カラーコードを定めるものではない。

- 同じ意味には同じ色を使う
- 同じ色を異なる意味へ使わない
- 一文の中で複数色を切り替えない
- 一画面の色数を増やしすぎない
- 色だけを唯一の識別手段にしない
- 暖色は警告表示に見えないよう使用箇所を限定する

完了条件：

- 各色の意味を説明できる
- 色を外しても文章構造が破綻しない
- 黒背景上で過度に強く見える箇所がない

### Pass 6：Disclosure / Folding

折り畳みは、読者が必要に応じて情報の深さを選ぶために使う。

判断基準は次の通り。

> その部分を読まなくても、本文の論理が成立するか。

折り畳み候補：

- 長い補足
- 詳細な具体例
- 数字や資料
- 出典一覧
- 本筋ではない余談
- 読者によって必要性が異なる情報

折り畳まない情報：

- 結論
- 論証に必要な前提
- 主要な反対意見
- 本文理解に必須の説明
- 記事の主題

利用可能な機能と構文は、
`04_LDS_Lodestone_Authoring_Reference.md` の検証状態を参照する。

完了条件：

- 折り畳みを開かなくても本文の論理が成立する
- 開閉位置とラベルから内容を予測できる

### Pass 7：BBCode実装

構造と視覚判断が固まった後に、Lodestone向け記法へ変換する。

- 04で利用可能性を確認した構文だけを使う
- 未検証構文を確定仕様として使用しない
- タグの入れ子と閉じ順序を確認する
- Markdown上の見た目をLodestone表示と同一視しない
- 変換によって本文の文言を変えない

完了条件：

- 使用した記法の根拠と検証状態が分かる
- タグの閉じ忘れや不正な入れ子がない

### Pass 8：実画面確認

Lodestoneの下書きまたは検証用記事で実表示を確認する。

- 見出し階層が明確か
- 本文より強調が目立ちすぎていないか
- 色が強すぎないか
- 空行が多すぎないか、少なすぎないか
- 長い行や見出しの折り返しが自然か
- 折り畳みの位置と操作が自然か
- リンクや埋め込みが正しく動くか
- PCとスマートフォンで破綻しないか
- 長文でも視線が迷わないか

確認できなかった環境や機能は、検証済みとして扱わない。

完了条件：

- 確認環境と未確認範囲を説明できる
- 表示上の問題が編集判断へ反映されている

### Pass 9：著者確認と最終推敲

視覚調整後の本文を、著者の意図と照合する。

- 主張の意味が変わっていないか
- 断定の強さが変わっていないか
- 引用と筆者の意見を混同していないか
- 語調と用語が統一されているか
- 導入と結論が対応しているか
- 誤字、脱字、表記揺れがないか
- 装飾によって意図しない重要度が付いていないか

内容判断を伴う変更は著者が確定する。

---

## 6. 各Passの成果物

各段階では、次の成果を確認してから次へ進む。

| Pass | 主な成果 |
|---|---|
| 0 | 編集条件と確認事項 |
| 1 | 論理構造と情報分類 |
| 2 | 段落構成と記事順序 |
| 3 | 見出し構造 |
| 4 | 文字階層と強調箇所 |
| 5 | 色の意味対応 |
| 6 | 折り畳み候補と本文必須情報の分離 |
| 7 | Lodestone向け実装 |
| 8 | 実表示の確認結果 |
| 9 | 著者確認済みの最終稿 |

成果物は必ず独立したファイルにするものではない。
記事草稿、編集メモ、差分、Case Studyなど、作業に適した形で保持する。

---

## 7. 参考記事の扱い

参考記事は、記事全体を模倣せず、部品と判断理由を分けて評価する。

確認項目：

- 参考URL
- 参考にする部分
- 解決している問題
- 採用する要素
- 採用しない要素
- 他の記事へ一般化できるか
- Lodestone上で再現できるか
- 実表示上の注意点

単一記事で使われている表現を、そのままLDS標準へ採用しない。

標準への採用条件と証拠の扱いは、
`01_LDS_Governance.md` に従う。

## 8. 編集完了チェックリスト

- [ ] 記事の目的と中心命題が明確
- [ ] 著者の主張、価値判断、結論を保持している
- [ ] 前提、根拠、分析、結論の関係が追える
- [ ] 見出しだけで記事の概要が分かる
- [ ] 段落ごとの役割が明確
- [ ] Typography、太字、色、余白の役割を説明できる
- [ ] 同じ意味へ同じ視覚表現を使っている
- [ ] 折り畳みが本文理解を妨げていない
- [ ] 使用した記法が04の検証状態と矛盾しない
- [ ] Lodestone実画面で確認した
- [ ] 未確認事項を確定扱いしていない
- [ ] 内容判断を伴う変更を著者が確認した

## 9. Case StudyとLDSへの還元

個別記事への適用結果は、方法論本文へ直接蓄積せず、
必要に応じてCase Studyとして分離する。

編集後に得られた知見は、次のいずれかへ還元する。

- 編集手順や判断基準：本書
- 設計判断の価値基準：`02_LDS_Design_Principles.md`
- 技術仕様、構文、制約、検証結果：`04_LDS_Lodestone_Authoring_Reference.md`
- 変更、証拠、昇格、廃止の規則：`01_LDS_Governance.md`
- 個別記事での判断と結果：Case Study

Case Studyの必須項目と知見の昇格条件は、
`01_LDS_Governance.md` に従う。
